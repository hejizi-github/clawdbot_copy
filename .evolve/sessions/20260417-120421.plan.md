# Session 20260417-120421 Plan

## Goal
Add SQLite-based result storage to trajeval — a core infrastructure component listed in the architecture diagram but never implemented beyond calibration annotations.

## Why This Is The Most Impactful Next Step
1. The proposal architecture shows SQLite storing "traces • scores • baselines • calibration data" — currently only calibration data uses it
2. Without persistent storage, users must manually manage JSON files to track history or run `improve`
3. This unblocks future features: trend analysis, automatic regression detection across runs, CI dashboards
4. The review explicitly said to push forward with core feature development

## Plan

### Files to Create
- `trajeval/storage.py` — `ResultStore` class wrapping sqlite3
  - Schema: `eval_results` and `judge_results` tables
  - Methods: `save_eval()`, `save_judge()`, `list_evals()`, `list_judges()`, `get_eval()`, `get_judge()`
  - Each row stores: trace_id, agent_name, timestamp, overall_score, passed, full JSON blob
- `tests/test_storage.py` — unit tests for all storage operations

### Files to Modify
- `trajeval/cli.py` — add `--store` flag to `eval` and `judge` commands, add `history` subcommand
- `trajeval/__init__.py` — export new module if needed

### CLI Additions
- `trajeval eval <trace> --store [path]` — save result to SQLite after evaluation
- `trajeval judge <trace> --store [path]` — save result to SQLite after judging  
- `trajeval history [--db path] [--limit N] [--agent name] [--failed-only]` — list past results
