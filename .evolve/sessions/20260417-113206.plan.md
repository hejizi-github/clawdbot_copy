# Session Plan: SQLite Storage Layer

## Goal
Implement the SQLite storage module listed in the architecture but never built — enabling persistent evaluation history, baseline comparison, and calibration data across runs.

## Context
- The proposal's architecture diagram shows "Storage (SQLite): traces • scores • baselines • calibration data" as a core layer
- Currently everything is in-memory; `compare` requires both traces to be passed as arguments each time
- No historical tracking is possible
- This is the most impactful missing piece from the original design

## Plan
1. Create `trajeval/storage.py`:
   - `TrajevalDB` class wrapping sqlite3
   - Tables: `eval_results` (trace_id, agent_name, timestamp, overall_score, metrics_json, metadata_json)
   - Methods: `save_eval(report)`, `load_eval(trace_id)`, `list_evals(agent_name, limit)`, `get_latest_baseline(agent_name)`
   - Default DB path: `~/.trajeval/history.db` (configurable)
   - Auto-create tables on first use

2. Create `tests/test_storage.py`:
   - Test save/load roundtrip
   - Test list with filtering
   - Test latest baseline lookup
   - Test duplicate handling (upsert)
   - Test empty DB edge cases
   - Use tmp_path fixture for isolation

3. Wire storage into CLI `eval` command (optional `--save` flag):
   - `trajeval eval trace.json --save` persists the result
   - `trajeval compare --baseline latest` loads from DB
