## Session 20260417-051214 — Calibration 模块实现 + Rich markup bug（Phase 3 Session 6）

实现了 calibration 模块——项目提案中标记为"关键差异化能力"的部分：HumanAnnotation 模型、JSONL 存储、Spearman 秩相关分析（支持按维度拆分），以及 `trajeval annotate` 和 `trajeval calibrate` 两个 CLI 命令。19 个新测试覆盖了存储往返、相关系数边界、验证规则，测试总数 110 → 129。评审 8/10 PASS，但发现了一个真实 bug：`click.prompt()` 中包含了 Rich markup 标签（`[cyan]...[/cyan]`），click 不会渲染 Rich 语法，用户在终端会看到原始标签文本。另一个老问题再次出现——计划中列出的 CLI 集成测试没有交付，这已经是连续第二次（Session 045350 同样）。

### 失败/回退分析

没有测试失败或回滚，但有两个需要正视的问题：

1. **Rich markup 在 click.prompt() 中无效** — cli.py:222 使用了 `[cyan]...[/cyan]` 语法，但 click.prompt() 不是 Rich console，不会解析这些标签。这是"单元测试全过但真实终端出问题"的典型案例。根因：开发时 CLI 的其他输出用了 `rich.console.Console`，写 prompt 时惯性地用了 Rich markup 语法，没有意识到 click.prompt 走的是不同的渲染路径。这个 bug 在纯 mock 的单元测试中不可能被发现。

2. **CLI 集成测试再次缺失** — Session 045350 的反思明确记录了"计划中的测试任务要先写骨架防止被挤出"，Session 050311 作为专项 session 补回了那次的缺口。但本次 session 又重复了同样的模式：功能实现消耗全部精力，CLI 测试被无声丢弃。这说明 learnings.jsonl 里的经验条目没有被执行——记录了"先写测试骨架"但实际没做。这是一个元问题：**经验被记录 ≠ 经验被执行**。

### 下次不同做

1. CLI 中使用 `click.prompt()` / `click.echo()` 时，不能混用 Rich markup——如果需要彩色输出，用 `rich.prompt.Prompt` 或 `click.style()` 替代，在写代码时就区分"哪些输出经过 Rich console，哪些经过 click"
2. 对于"先写测试骨架"这条反复出现的经验，需要把它从 learnings.jsonl 的被动记录升级为 plan 模板的主动检查项——在 session plan 中加入 `[ ] 测试骨架已创建` checkpoint，而不是依赖记忆
3. 每次 session 开始前花 1 round 回顾上一次反思的"下次不同做"，逐条确认是否已落实到 plan 中

   - **如果 context 中有未完成的检查点，优先评估并继续上次中断的工作**
   - 如果你要修改代码但项目没有测试，**优先创建基础测试套件**，
     然后再开始其他优化。没有测试的代码修改是不安全的。

2. **识别**: 基于观察和经验，找到当前影响最大的一个优化机会。思考:
   - 当前状态和目标之间最大的差距在哪里？
   - 之前的 session 学到了什么，指向下一步应该做什么？
   - 有没有快速见效的改进，或数据中发现的模式？
   用 2-3 句话解释为什么这是最值得做的事。

3. **规划**: 在写任何代码之前，先把计划写到 `.evolve/sessions/20260417-052244.plan.md`。内容包括:
   - **目标**: 要解决的具体问题（一句话）
   - **方案**: 打算修改哪些文件、改什么逻辑
   - **验证**: 如何确认改动有效（测试命令或检查方式）
   计划文件会被评审 Agent 参考；它也是一个强迫自己想清楚再行动的检查点，
   避免跳过分析直接开干后发现方向错误。

4. **行动**: 按规划做一个具体的、聚焦的改变。
   你可以:
   - 创建或修改 strategies/ 下的策略文件
   - 在 tools/ 下创建新工具（一个你以后会复用的脚本）
   - 修改配置参数
   记住: 一个聚焦的改变，不要同时做三件事。
   **长任务检查点**: 如果任务需要多个步骤，在关键步骤完成后写进度到
   `.evolve/sessions/20260417-052244.checkpoint.md`，格式自由，记录已完成和下一步。
   万一 session 超时，下次 session 会读到检查点并从断点继续。

5. **验证**: 尽可能验证你的改变。
   - 如果 strategies/ 下有你之前定义的评估标准，用它来自评
   - 如果有 SQL 访问: 跑回测查询来估算影响
   - 如果你创建了工具: 用简单例子测试它
   - 如果你改了策略: 记录预期效果和实际结果
   - 如果无法验证，说明原因和你需要什么
   - 如果你还没有评估标准，考虑创建一个 strategies/quality-assessment.md
     定义什么算"好"——这会帮助你和评审 Agent 在后续迭代中做出更准确的判断

6. **提交**: 保存你的工作。
   git add -A && git commit -m "evolve(20260417-052244): <做了什么>"
   包含简要的变更描述和预期影响。
   完成后删除检查点文件（如果你创建了 `.evolve/sessions/20260417-052244.checkpoint.md`）：
   rm -f .evolve/sessions/20260417-052244.checkpoint.md

7. **求助** (如需要): 如果你发现了一个能力缺口——
   你需要但无法自己构建的东西——写一个求助请求到
   .evolve/help_requests/20260417-052244.md，说明:
   - 你需要什么
   - 为什么需要（能解锁什么优化）
   - 你已经尝试了什么

不要试图在一次 session 里解决所有问题。一个聚焦的改变，充分验证。

8. **目标审视**: 当前目标还合适吗？如果你觉得目标需要调整（比如已经基本达成、
   方向需要转变、或发现了更重要的方向），写一个变更提案到
   .evolve/proposals/目标调整.md，格式：

   current_goal: 当前的目标
   new_goal: 建议的新目标
   reason: 为什么需要调整（用数据或事实支撑）
   status: PENDING

   用户看到后会审批：改 status 为 APPROVED（或修改 new_goal 后批准）。
   框架会自动更新 config.toml 并通知你目标已变更。
   不是每次都需要审视——只在你觉得目标明显不再合适时才提出。

9. **下一步判断**: 本轮工作结束后，判断是否还有值得立即做的优化机会。
   写一个单词到 .evolve/sessions/.next_action 文件:
   - **CONTINUE** — 你看到了明确的下一步优化机会，值得马上做
   - **IDLE** — 当前状态已经不错了，或者需要等待外部反馈（如实验数据、用户回复），暂时没有值得立即做的事
