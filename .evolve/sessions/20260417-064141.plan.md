# Plan — Phase 3 Session 16: Compare output details + test cleanup

## 目标
Close the 4-session-old compare CLI recovery_window test gap by adding details to MetricDelta, then reorganize misplaced tests.

## 根因分析
The compare test can't verify `recovery_window` flows through because `MetricDelta` only carries `baseline_score`/`current_score` — no `details`. The eval output includes `details` per metric, but compare strips them. This is an output format asymmetry, not just a test issue.

## 方案

### 1. Add details to MetricDelta (compare.py)
- Add `baseline_details: dict | None = None` and `current_details: dict | None = None` to `MetricDelta`
- In `compare_reports()`, populate from `baseline_map[name].details` and `current_map[name].details`
- This makes compare output inspectable at the same level as eval output

### 2. Fix compare CLI recovery_window test (test_cli.py:256-266)
- Parse JSON output, extract `error_recovery` from `metric_deltas`
- Assert `baseline_details["recovery_window"]` == 1 and == 5 respectively
- Mirror the eval test pattern (test_cli.py:77-97)

### 3. Move latency_budget integration tests (test_metrics.py:538-569)
- Move `test_evaluate_includes_latency_budget`, `test_config_latency_budget_flows_through_evaluate`, `test_config_latency_budget_default_is_none` from `TestErrorRecovery` to new `TestLatencyBudgetIntegration` class

### 4. Add unit tests for MetricDelta details
- Test that compare_reports populates baseline_details and current_details
- Test that details are null when metric has no details

## 验证
- `python -m pytest tests/ -q` — all 210+ tests pass
- New tests verify MetricDelta details carry through
- Compare CLI recovery_window test actually asserts recovery_window values (not just exit code)
