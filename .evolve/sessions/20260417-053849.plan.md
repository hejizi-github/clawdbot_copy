# Session Plan: End-to-End Integration Tests

## Goal
Add integration tests covering the full eval → judge → calibrate pipeline, addressing the reviewer's #1 recommendation and the "测试覆盖 ↑" KPI.

## Why this is the highest-impact change
- Reviewer explicitly said: "建议优先做集成测试——目前 session_metrics 显示 test_count 一直为 0，而考核指标明确包含'测试覆盖 ↑'"
- Existing 137 tests are unit-level; judge CLI tests mock `judge()` entirely
- README promises specific behaviors (exit codes, JSON output format) that are only partially verified
- Integration tests catch interface contract breakage between modules

## Plan

### New file: `tests/test_integration.py`

**Test 1: Full deterministic eval pipeline (Python API)**
- Create a trace dict → `ingest_json()` → `evaluate()` → verify EvalReport fields
- No CLI, no mocks — pure module integration

**Test 2: Eval → Compare cross-command pipeline (CLI)**
- Run `eval` on two fixtures via CliRunner → collect JSON → run `compare` → verify regression detection
- Tests the real CLI → ingester → metrics → compare chain

**Test 3: Judge with fake Anthropic client**
- Create a mock client that returns valid JSON (not mocking `judge()` itself)
- Call `judge(trace, config, client=fake_client)` → verify JudgeResult parsing
- This exercises `build_user_prompt` → API call → `_parse_response` → `_normalize_score`

**Test 4: Full calibration pipeline**
- Use fake judge results + annotations → `compute_correlation()` → verify CalibrationResult
- Chain through AnnotationStore save/load to test JSONL round-trip

**Test 5: CLI eval JSON → judge JSON → calibrate JSON pipeline**
- Simulate the documented CI workflow: eval produces report, judge produces judgments, calibrate correlates
- Uses CliRunner chaining with tmp files

**Test 6: README's Python API example (runnable)**
- Reproduce the README code snippets as an actual test to verify documentation accuracy
