# Plan: SQLite Result Storage Layer

## Goal
Implement SQLite-backed persistence for evaluation results — the missing MVP Session 2 feature that unlocks historical comparison, baselines, and trend analysis.

## Rationale
- Reviewer explicitly says: stop test maintenance, push core features
- SQLite storage is in the original proposal (Session 2 target) but not implemented
- It's foundational infrastructure: without persistence, every eval is ephemeral

## Changes

### 1. New module: `trajeval/storage.py`
- `ResultStore` class wrapping SQLite
- Tables: `eval_results` (trace_id, overall_score, passed, metrics_json, timestamp, agent_name, config_json)
- Methods: `store_eval()`, `get_history()`, `list_results()`, `get_result()`
- Auto-creates DB file on first use

### 2. CLI integration: `eval --store <db_path>`
- Add `--store` option to `eval` command
- After evaluation, persist result to SQLite
- Print confirmation message

### 3. New CLI command: `trajeval history`
- `trajeval history --db <path>` — list stored results
- `--trace-id` filter, `--limit`, table output via Rich

### 4. Tests: `tests/test_storage.py`
- 10+ unit tests for storage CRUD
- Test CLI `--store` integration
- Use tmp_path fixture for isolated DB files
