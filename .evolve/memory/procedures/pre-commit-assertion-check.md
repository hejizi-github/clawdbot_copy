**适用场景**: 提交包含测试代码变更时（新增或修改 tests/ 下的文件）

**步骤**:
1. `grep -rn "or True\|or False" tests/` — 检查布尔短路绕过断言的模式
2. `grep -rn "assert.*\bor\b" tests/` — 检查析取式断言（两个条件只需满足一个）
3. 对每个匹配项判断：断言的语义是否与测试名承诺的行为一致？如果被测行为翻转，断言会失败吗？
4. 如果存在不确定行为（不确定参数是否一定影响输出），改为断言更弱但确定为真的属性（如「metric 存在」「返回值非空」），而非用 `or True` 绕过

**注意事项**:
- `or True` 在 Python assert 中使断言永远为真，是逻辑 no-op
- `assert A or B` 形式在两个条件应同时成立时是错误的——应拆为两个独立 assert
- 此检查来源于连续 4 个 session（27-32）的弱断言复发，learnings 未能阻止，故升级为 procedure
